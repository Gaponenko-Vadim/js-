# LECTURE_CREATION_GUIDE 2.0 — живые лекции, которые хочется читать

Коротко: это обновлённые правила создания лекций **в живом стиле** — с объяснениями «что это и зачем», аналогиями, примерами, ошибками и практикой. Стиль ориентирован на лекцию **«HTML: формы — основы»** в актуальной версии и на лучшие части лекции **«OAuth 2.0 и JWT»**.

Цель — чтобы после чтения студент **понимал**, а не просто «знал факт», и мог **применить** знания на практике.

---

## 🎯 Зачем это нужно

- Сделать лекции **понятными, живыми, практичными**.
- Закрывать **часто задаваемые вопросы** прямо в тексте.
- Давать студенту ощущение: «я реально понял и могу применить».

---

## 🧭 Базовая модель лекции (обязательно)

- `title` — название лекции
- `topic` — тема/раздел
- `content` — основной текст (вкладка «Лекция»)
- `scenariosContent` — сценарии (вкладка «Сценарии»)
- `exampleContent` — пример (вкладка «Пример»)
- `tasksContent` — задания (вкладка «Задания»)

---

## ✅ Принципы стиля (обязательные)

1) **Простыми словами** — объясняем «как человеку», без академического занудства.
2) **Поясняем «зачем»** — не просто что это, а почему и когда это нужно.
3) **Аналогии из жизни** — формируют понимание, а не зубрёжку.
4) **Проблема → решение** — показываем контекст, как в бизнес‑задаче.
5) **Мини‑выводы** — после каждого блока: «✅ Вывод: …»
6) **Ошибки и best practices** — разбираем типичные провалы и правильный путь.
7) **Сценарии + практика** — отдельные вкладки для закрепления.
8) **Interview‑FAQ** — в лекции обязательно упоминаем темы, по которым чаще всего спрашивают на собеседованиях.

---

## 🧩 Структура основного текста (content)

> Эта структура должна ощущаться одинаково читаемой во всех лекциях.

### 1) Введение (10–15%)

- Простая аналогия.
- Зачем тема нужна в реальной жизни.
- Итоговый вывод.

**Пример структуры:**

```markdown
# [Название темы]

## Введение: [аналогия из жизни]

[2–3 предложения про проблему/контекст]

💡 Совет: ...
✅ Вывод: ...
```

---

### 2) Проблема → решение (5–10%)

- «Без этого — хаос»
- «С этим — структура»

**Важно:** это создаёт мотивацию читать дальше.

---

### 3) Ключевые термины (обязательно)

Читателю нужен «словарик».

**Шаблон:**

```markdown
## Ключевые термины (простыми словами)

- **Term (перевод)** — короткое объяснение.
- **Term (перевод)** — короткое объяснение.
```

✅ **Правило перевода:** при первом упоминании английского термина обязательно даём перевод в скобках.

---

### 4) Ключевые элементы (50–60%)

Каждый ключевой элемент раскрываем одинаково, чтобы чтение было лёгким.

**Шаблон блока:**

```markdown
## N. [Элемент] — [смысл]

**Назначение:** ...
**Аналогия:** ...

**Пример:**
```html
...
```

**Характеристики:**
- ✅ ...
- ✅ ...

**Когда использовать:** ...
✅ **Вывод:** ...
```

Минимум: **Назначение → Пример → Когда использовать → Вывод**.

---

### 5) Сравнения и таблицы (5–10%)

- Таблица: «A vs B»
- Упрощает запоминание.

---

### 6) Типичные ошибки (5–10%)

**Формат:**

```markdown
### Ошибка N: [Название]
❌ Неправильно: ...
✅ Правильно: ...
**Почему:** ...
```

Минимум 5–8 ошибок.

---

### 7) Best Practices (5–10%)

Короткий список «делай так». Без воды.

---

### 8) Заключение (3–5%)

- 2–4 ключевые мысли
- Мотивирующая фраза

---

## 🎬 Сценарии (scenariosContent)

**Цель:** показать реальные кейсы и решения.

**Обязательно:**
- В начале — **«Типичные ошибки (сводка)»** списком.
- 5–15 сценариев.
- Формат **До/После** + пояснение.

**Шаблон:**

```markdown
## Популярные сценарии

### Типичные ошибки (сводка)
- ...
- ...

---

### Сценарий 1: [Ситуация]

❌ Неправильно: ...
✅ Правильно: ...

**До:**
```html
...
```

**После:**
```html
...
```

**Почему**: ...

**Практический пример:** [фрагмент требования]
```
...
```
```

---

## 🧾 Пример (exampleContent)

**Обязательный стартовый контекст — строго по шаблону:**

```
Разрабатывается условно такой сайт: платформа онлайн‑курсов.
После всех митингов с бизнесом и постановки задачи нужно было решить проблему:
пользователи терялись в каталоге и не понимали, где они находятся.

Ниже — фрагмент готового документа по [теме лекции].
```

**Дальше обязательно:**
- **Цель**
- **Требования**
- **Критерии приёмки**
- При необходимости — короткий фрагмент кода

**Шаблон:**

```markdown
## Пример: [Название]

[обязательный контекст]

---

### Раздел: [Название]

**Цель:** ...

**Требования:**
1) ...
2) ...

**Критерии приёмки:**
- ...

**Фрагмент реализации (если нужно):**
```html
...
```
```

## 🧪 Задания (tasksContent)

**Главное правило:** все задания одной лекции — **один общий проект**.

### Обязательный блок «Подготовка к заданиям»

Что должно быть:
- 🎯 **Цель**
- 📘 **Контекст проекта**
- 🧩 **Как работать**
- **Каркас проекта** (полный HTML, а не кусок)

**Пример структуры:**

```markdown
## Подготовка к заданиям

🎯 **Цель:** ...
📘 **Контекст:** ...
🧩 **Как работать:** ...

Каркас страницы:
```html
<!doctype html>
<html lang="ru">
  <head>...</head>
  <body>...</body>
</html>
```
```

### Формат задания (обязательно)

```markdown
### Задание N: Название
**Кратко:**
Ситуация: ...
Что сделать: ...
Критерий: ...

**Описание:**
🔎 **Ситуация:** ...
✅ **Что сделать:** ...
📌 **Требования:**
- ...
🎯 **Критерий:** ...

🧱 **Исходник (как сейчас):**
```html
...
```

**Ответ:**
**Было:**
```html
...
```

**Стало:**
```html
...
```

**Объяснение:** 2–4 предложения (почему так, что улучшилось, как проверить).
```

**Количество заданий:** 7–15.

## ✍️ Тон и подача

✅ Используем:
- короткие предложения
- простой язык
- аналогии
- мини‑выводы
- эмодзи для навигации

❌ Избегаем:
- сухих определений без контекста
- длинных абзацев
- «академических» формулировок

**Ориентир:** 10–15 слов в предложении.

## 🎤 Обязательное покрытие Interview‑FAQ

В каждой лекции должны быть явно упомянуты **темы, которые часто спрашивают на собеседованиях** по этой теме.  
Это может быть отдельный абзац, мини‑блок «Часто спрашивают» или встраивание в основные разделы.

**Как делать правильно:**
- выбираем 5–10 самых популярных вопросов по теме
- объясняем их в тексте (не списком «вопрос‑ответ»)
- даём короткий вывод «что важно запомнить»

✅ **Цель:** студент должен уверенно отвечать на интервью‑вопросы после чтения лекции.

---

## 🚫 Частые ошибки при написании лекций

- Лекция = список фактов без объяснений
- Нет «зачем это нужно»
- Нет примеров или сценариев
- Нет связанной практики
- Нет «Подготовки к заданиям»

---

## 💡 Золотая формула

**Проблема → объяснение → пример → вывод → практика**

Если лекция следует этой формуле — она будет понятной, живой и полезной.
